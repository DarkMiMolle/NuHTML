1 - Inheritance in the structure world

extension allows to add methods to a type. With structures, it is also possible to "add" attribute to an existing structure. Actually it creates a new type like we see earlier, and we append attribut to it.

package main 

struct type B = {
	b int = 42
}
struct type A = B with {
	c int
} && default = {{b: 18}}

func main() {
	a := A{} // default constructor
	b := a as B // ok ! A comes from B !
}

So it is not directly an extension of the type B, but A can be explicitly cast as a B ! Moreover, pointer will keep the dynamic type, so it could be possible to re-cast it to A ! The syntax is not the same as interfaces, because it is more close to the typedef. It is not possible to "inherits" from multiple structures.

package main
import Std io.console as cio

struct type Base = {
	get Value int
}
struct type A = Base with {
	get Prev int
}
struct type B = Base with {
	get Next int = .Value
}
extension type Base = {
	set Value(v int) = default // .Value = v
	final const Print() => cio.Print("Base not matter what")
	// final indicate it won't be override
}
extension type A = {
	
	override set Value(v int) {
		.Prev = .Value
		.Value = v
	}
	// override the set Value of Base if the dynamic type of a pointer is A !

	final const Print() => cio.Print("A no matter what")
	// not an override of Print, juste a new one specific to A that won't be override neither.
}
extension type B = {
	set Value(v int) {
		.Value = .Next
		.Next = v
	}
	// same with B, override is missing, but it will juste generate a warning lv 1
	const Print() => cio.Print("B but will depend on 'sub-type'")
	// not an override of Print, juste a new one specific to B, that may be override in this sub-type !
}

func main() {
	var base *Base
	if t := cio.Get<string>(); t == "A" then base -> new A as *Base
	else if t == "B" then base -> new B as *Base
	else then base -> new Base
	// not an if-exp, the -> is for the pointer !

	base.Value = 42
	if base is *A then cio.Println(base.Prev) // output 18
	else if base is *B then cio.Println(base.Next) // output 42
	else then cio.Println("base is Base")

	base.Print() // will display "Base no matter what", even if the dyn type is A or B
}

There is something interesting too, we see here. The method set Value set Value(v int) can be defined as default in the extension (juste like how it works with interfaces), but can be re-defined in the new created types !! And during the execution, it will call the set method of the dynamic type !! This is called the dynamic dispatch. It is possible to avoid that behaviour (and then call the set method of Base even though the dynamic type of the pointer is A or B). To do so, we have to put final before the method name (or the const if any). It indicates the method won't use the dynamic dispatch.

Note for Pointer: Here, we use the if-is statement, that we first saw during the chapter on the interfaces. With pointer it is also really interesting to use it. Pointer are considered like references and then, there type is actually the final type of the data the point to. But with an is expression, we can put both a pointer-type or a non-pointer-type ! If the pointer (call it: ptr) is type: **int, both: ptr is int and ptr is **int will return true ! But it will be false with *int. Even more interesting. Pointers keep the dynamic type of the address they hold. So it is possible to use is with a super or sub pointer type ! In the code example, base is a pointer, so base is Base works and is always true, base is *Base too, base is A will always be false, but, base is *A will depend on the dynamic type hold by the pointer ! That's why it is possible to use the if-is with pointer ! And here, A and B are more close to the real type hold by base so in the if scope, base is considered to be A (or B if we are in the is *B).

2 - Create non-exported attributs

We saw the attributs of a structure are always public (visible from everywhere). It is possible to define attributes visible only inside the current package/namespace of the structure definition. Instead of defining the structure type, we define the structure (local) implementation. The implementation is separate from the definition of the type, since it is only useful in the package/namespace scope.

// agreed we are in the package Pkg1 in the file: file1.nu
package Pkg1

struct type A = {
	C *int
	
	init(c int) // if no definition, "implem" block is required
} 
struct implem A = { // implem are never exported
	c int
	init(c int) = default {
		C -> &.c
	}
}
// otherFile.nu in the same pkg
package Pkg1
structure := A{}
val := structure.c // ok, it is accessible inside the package

// otherFile.nu in package Pkg2
package Pkg2
import Pkg1
structure := A{}
// structure.c // ko --> not exported
val := structure.C // ok, it is public

There is more. We saw we could "inherits" from structure. But attributes defined in the implem block are only visible for the current package/namespace. What if we want to let the opportunity for the sub-type to use an attribute that wouldn't be visible from out of the new structure ?

package Pkg1
struct type A = {
	C *int
	init(c int)
	init(C *int)
	internal c int  // only visible if there is a sub-type !
} with extension {
	const Val() => .val //  ≈ get Val string
}
struct implem A = { // not visible at all from out of the package
	val string = "private"
	init(c int) = default {
		.C -> &.c
		val = "still private"
	}
}

func DisplayVal(a A) => a.val // ok, still in the same package


package Pkg2
import {
	Pkg1
	Std Strings as Str
}

cast from string to int is explicit -> Str

struct type B = A with {
	val string = "public" // val from A is not visible !!
}

func Display_c(b B) => b.c as string // ok, it is the c of B !
// if it was (a A) => a.c as string it would have failed because c is internal !

When using the implem block, the init using the attribute names (exp: A{C: new int, c: 42, val: "coucou"}) can initiate the internal and implem attributes only in the current namespace. Else those are not visible, so it would be: A{C: new 42} and other value would be defaulted. That's why it is recommended to set default value to not public attributes. By the way, it is possible to combine internal and get as much as it is possible to use get attributes in an implem block. Top | Bottom

3 - With interface

With the interface it works the same way. However, the external implem block are not possible. Also, as internal is visible only in the current package/namespace, if a type from another package/namespace implement an interface with internal method, then the internal method is not accessible if the type of the value is the interface.

package Pkg1

interface type A = {
	const Print()
	internal const msg() string
}


// other file
package Pkg2
import {
	Pkg1 as _
	Std io.console as cio
}

type MyInt = int with extension
public alias mi -> MyInt

extension type MyInt: A = {
	const Print() => cio.Printl(.msg)
	internal const msg() => "Hello there"
}

func DisplayInfo(MyInt int) => cio.Print(i.msg) // ok, internal in the same pkg
func DisplayInfo(a A) => a.Print() // ok, can't use the .msg of a because it is internal to the A interface and we could not own the type !

//other file
package main
import {
	Pkg1 as _
	Pkg2 as _
	Std io.console as cio
}


func main() {
	var a A = 42 mi // ok MyInt implement the interface A
	a.Print() // print: Hello there
	// the method 'msg' is not accessible because it is part of the internal A interface
}

4 - A new way to define cast: cast-init

With structures it is possible to define init status. That init "method" (the constructor) can allows us to build the structure with another type in input ! That behaviour is really similar of creating a cast from the type to the structure. That's why it is possible to let Nu define the cast to a structure, if it has an initializer who takes only the given type !

package main

import Std io.console as cio

struct type Complex = {
	get re, im fraction
	init(val int) {
		.re = val
	}
	init(re fraction) = default
	init(re, im fraction) = default
}

cast int as Complex is implicit // OK !!
/* equivalent to
cast int as Complex {
	cast.to = Complex{cast.from}
} is implicit
*/

func main() {
	var v Complex = 42 // ok ! call the implicit cast !!
	cio.Println(v) // output: Complex{re: 42, im: 0}
}
