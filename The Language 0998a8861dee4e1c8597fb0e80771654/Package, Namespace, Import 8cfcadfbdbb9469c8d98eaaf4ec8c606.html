1- Packages & Namespace

1.1 - packages

Every code in Nu must be in a package. The package structures the code and the importations. In a project, a package must be represented by a directory of the same name and with the extension .pkg.nu; all code written in that directory belongs to the package of the same name. It is not possible to add element to a package if we are not currently in that package. A package in another package must also be represented by a directory of the same name. However, instead of naming the package simply with the name of its directory, it must be the path from the first package, where '/' are '.' Example.nu.project/Package1.pkg.nu/Package2.pkg.nu/file.nu

// file.nu; Package1.pkg.nu/Package2.pkg.nu --> Package1.Package2
package Package1.Package2

// code

Inside a package every declaration starting with uppercase is exported except if the keyword private is used before the declaration, and every declaration starting with lowercase are, in opposite only accessible from inside the package. A package can also be private. If a top folder package is private, it means it won't be visible by other project. namespace declaration are particular. Namespace are exported by default no matter what name they have. To make a namespace private, we also need to add that keyword before the declaration. Finally, there is a last degree of visibility whom allows to be private to the current file: fileprivate. It is not possible to make a namespace as fileprivate. example: Top | Bottom

// agreed that we are in the package Pkg1 in file: file1.nu
package Pkg1

namespace NP1 {
	const val = 42 // not exported from out of the namespace
	const Val = val // exported
}

const Val = 42 // exported
private const Val2 = 18 // only visible inside Pkg1
fileprivate const val = Val // only accessible inside that file 

private namespace NP2 { // only visible inside Pkg1
	const val = 42 // only visible inside the namespace
	const Val = val
}

// agreed that we are in the package Pkg2 in file: file2.nu of the same project
package Pkg2
import Pkg1

const my_val = Pkg1.NP1.Val // ok, my_val is not exported
// my_val = Pkg1.Val2 // ko, Val2 is declared private

1.2 - Namespaces

Namespace are a way to structure code like package. However, unlike package, namespace, doesn't need to be represented by a directory. And what's really makes the difference is that namespace can be extended from everywhere and automatically import the whole namespace. But, when extending a namespace from out of its package location, it won't be possible to access the namespace-protected elements.

1.2.1 - Create one

To create a namespace, we simply need to use the keyword: namespace, followed by its name. While in the same package, a namespace can be "created" more than one time. It is actually extended, so duplicating code will be a compiler error. Let's see example:

// agreed that we are int the package TypesAddon in the file file1.nu
package TypesAddon

type MyInt = int // exported

namespace Number {
	const! pi = 5419351 \ 1725033 // pi approximate at 10^-10
	func Square(v MyInt) MyInt => v * v // exported, use of MyInt from package TypeAddon
}

//agreed that we are in the package TypesAddon in the file file2.nu
package TypesAddon

namespace Number { // extend the existing one
	func! PI() fraction => pi // ok, can use pi since it is extended in the same package
}

// agreed that we are in the pacakge main in the file main.nu

package main

import {
	TypesAddon as ta
	Std io.console as cio
}

namespace ta.Number { // exporting the namespace TypesAddon.Number !!
	func Square(v fraction) fraction => v * v
}

func main() {
	cio.Println(ta.Number.Square(ta.Number.PI()))
	// print: 29 369 365 261 201 \ 2 975 738 851 089
}

1.2.2 Extend namespace

It is possible to extend a namespace. But it has to be exported. To do so we simply mark the full name of the namespace from the top package. If we are in a sub-package, there is no need to import anything, but if not the package that contains the namespace must be imported. Remarque: it is better to extend a namespace over creating a new one.

1.2.3 Import extended namespace

To use extended namespace from another package, the package that extend the namespace has to be imported in addition of the base namespace.

2- Import

There are 3 kind of importation in Nu. But every import are preceded by the keyword import. That keyword must appear only once in a file, so if there is multiple import, they must be surround by '{}'. It is possible to import a package, or a namespace, and it is possible to rename them (it is even mandatory if 2 have same name)

2.1 - Import from the same project

if we are in a project, it is possible to import element from the same project, that may be in a different package. To do so, simply write the package name.

package aPkg
import AnotherPkg

// code

Then the whole package is imported, and to use exported element from 'AnotherPkg' we must write: AnotherPkg.EXPORTED ELEMENT. It is possible to avoid that usage.

package aPkg
import AnotherPkg as _

// code

as allows to use a different symbol to refer what is imported. Here, we say as _, which means, there is no need to use a symbol, you can directly use the exported element from "AnotherPkg".

2.2 - Import from another project

In a project or not, it is also possible to import element that are in another Project. First the project must be refered, then what we want to import.

package aPkg
import {
	Std io.console as cio
	"https://www.github.com/NuSample/GitImport" pkg.ns
}

// some code

Here "Std" is the Standard Project of Nu, it is on your computer with the nu-cmd install. If a project is on your computer, nu-cmd should be able to find it, then there is only need to put the project name. If 2 projects have the same name or if a project isn't on the computer, an url must be given in literal string. The string may start with "http", "https", "file://" or "relative/" http(s) must be an url to a git repository with the '.nu.project' or '.nu.config'. "file://" is the file protocol (file://host/absolute/path, or if host is localhost: file:///path), "relative/" is the path to the .nu.project or .nu.config with relative path.

Top | Bottom

2.3 - Import from file (non project)

Finally, it is possible to import file by themselves. It is not encourage to do so, but you may want to load a script file. To do so, you must specify the project name as '_' and put the path in literal string like 2.2 part, but it is mandatory to "rename" them.

package aPkg
import {
	_ "https://www.github.com/NuSample/ScriptGitImport/master/script.nu" as script
}

// some code

The path must lead to the file itself.
