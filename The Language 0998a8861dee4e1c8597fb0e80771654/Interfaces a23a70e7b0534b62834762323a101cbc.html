1 - Implement an interface

An interface is a groupe of function that can be implemented by a type. The interface can be used at a type who owes a value that implement all its method. It is even possible to ask an init function ! An interface can be specifically implemented through an extension, then the one type can be implicitly cast to the interface, but if a type implement all method of the interface without specifically saying it implement the interface, then the cast must be explicit.

package main

import Std io.console as cio

interface type Number = {
	const Square() Self
}

extension type int: Number = {
	// the Number methods MUST be implemented
	const Square() Self => self * self
}

extension type float = {
	const Square() Self => self * self
}

func main() {
	var n Number = 18 // ok
	cio.Println(n.Square) // the only method n has.
	n = 31.08 as Number // n now holds a float !!
	if n is float {
		n += 1 // only possible because n is considered as a float here !
		cio.Println(n)
	}
	else if n == 18 then cio.Println(18)
// n == 18 is not equal to n == 18. nor n == 18.()
// first n is int then == 18
// second, n is float then == 18.0
// third, n is fraction then == 18.(0) (aka 18 \ 1)
}

We don't need to specify func in an interface, since it can not be other things than functions. By the way, the default value of an interface is nil. Top | Bottom

2 - interface extension

It is possible to extend an interface ! So if a type implement the interface, it won't have to implement functions extended (but can if wanted); it's like giving default implementation for methods.

package main

import Std io.console as cio

interface type Printable = {
	const Print()
}

extension type Printable = {
	const Println() {
		.Print()
		cio.NewLine()
	}
}

extension type int: Printable = {
	// Print MUST be implemented, but Println can not be.
	const Print() => cio.Print(self)
	const Println() -> Printable  // as it is extension, it stil needs to be reminded
	// we could have written: -> default since the is only one "super"
}

extension type string = {
	const Print() => cio.Print(self) // enought to be considered like a Printable
}

func main() {
	42.Println()
	var p Printable = "Hello World !" as Printable
	p.Println()
	
}

3 - Multiple interface and inheritance

3.1 - inherits from other interface

It is possible to inherits from other existing interface.

package main

interface type A = {
	A()
	AB()
}

interface type B = A && {
	B()
	AB()
}
// AB can be duplicate it doesn't matter here...

struct type C = {}

extension type C: B = {
	A() {}
	B() {}
	AB() {} // AB must appear only once (it is the same for A and for B)
}

func main() {
}

It is possible to inherit multiple interface.

package main

import Std io.console as cio

interface type A = {
	A() string
}
extension type A = {
	const Println() => cio.Println(.A)
}
interface type B = {
	B() string
}
extension type B = {
	const Println() => cio.Println(.B)
}

extension type int: A & B = {
	A() => "A"
	B() => "B"
	// also mandatory
	const Println() => cio.Println(.A, .B)
	// or const Println() -> A
	// or const Println() -> B
}

func main() {
	var a A = 42
	a.Println() // it will print: A B because Println is overrided for int !
}

3.2 - Multiple interface type

It is possible to use value we expect to follow some behaviour. That's why it is possible to have multiple interface type.

package main

interface type AquaLiving = {
	Swim()
	const IsAlive() bool
}

interface type GroundLiving = {
	Walk()
	const IsAlive() bool
}

struct type Crocodil = {
	get IsAlive = true
	get Position struct{x, y int}
}

extension type Crocodil = {
	Swim() => .position.y++
	Walk() => .position.x++
	const IsAlive() bool -> Crocodil
}

alias AquaGroundLiving -> (AquaLiving & GroundLiving)
// also possible: interface type AquaGroundLiving = AquaLiving & GroundLiving & {}

func main() {
	var animal (AquaLiving & GroundLiving) // AquaGroundLiving
	animal = Crocodil{} as AquaGroundLiving
}

We see that Crocodile doesn't implement the method IsAlive. It's because a get attribute can be considered like a function of the same name returning itself get IsAlive = true <=> __nu_IsAlive bool; func IsAlive() => .__nu_IsAlive The most simple in that kind of situation is to make Crocodile inherits from both interface. 3.3 - Already existing values

if a type already have data of the same name the interface, but they are not compatible, then it is not possible to make its type implement the interface.

package main

interface type A = {
	Data() string
}

struct type B = {
	Data int
}

// B can't be extended with A.

func main() {
}

Here B.Data is int, so it is not possible to match A.Data. Top | Bottom

4 - the set functions

With attributes we had the get, that allowed us to get an attribute but not changing it, we saw something similar with getter methods. Now it's time to see the setter methods. Methods that have the prefix set will be usable like variable assignments. A getter is mandatory to make them works (or, a public attribute of the same name), and the prototype of those function must be: func(T)void where T is the type returned by the getter.

package main

import Std io.console as cio

interface type Settings = {
	const Size() {{w, h int}}
	set Size(s struct{w, h int})
}

struct type WindowInfo = {
	Size struct{w, h int}
}

func main() {
	var s Settings = WindowInfo{} as Settings 
	s.Size = {{w: 2800, h: 1800}}
	cio.Println(s.Size)
}

Here Size in the interface match Size in the structure, so implicitly it can be implemented. It is possible to make it implement the interface anyway:

package main

import Std io.console as cio

interface type Settings = {
	const Size() {{w, h int}}
	set Size(s struct{w, h int})
}

struct type WindowInfo = {
	Size struct{w, h int}
}
extension type WindowInfo: Settings = {
	const func Size() -> WindowInfo
	set func Size(s struct{w, h int}) -> WindowInfo
}

func main() {
	var s Settings = WindowInfo{}
	s.Size = {{w: 2800, h: 1800}}
	cio.Println(s.Size)
}

5 - if is statement

The interfaces variables always have 2 kind of type. The static one, compute at compile time, is the interface type, and the dynamic one, compute at runtime, is the "real" type holds by the interface variable. It is possible that we want to acte specifically weathers the interface holds a type or another. The if-is allows to check if a value is a certain type. It is based on the is operator that return a boolean. Nothing new here. However, using if-is on a variable will also make something more. If the is operator return true, and if the type tested is "more precise" (match more the dynamic type), then the variable will be considered like a variable of that type ! Let's see an example, but we'll talk about that later in the chapter : Type & dynamic type

package main

import Std io.console as cio

interface type GeneratorValue = {
	Generator() *Generator
}

interface type Generator = {
	InitValue()
	Next() GeneratorValue
	const IsEnded() bool
}

extension type int: Generator && GeneratorValue = {
	InitValue() {
		self = 0
	}
	const Next() GeneratorValue {
		self++
		return self
	}
	const IsEnded() => false
	
	Generator() => &self

	SelfDouble() => self *= 2
	
}
extension type float: Generator && GeneratorValue = {
	InitValue() {
		self = 0
	}
	const Next() GeneratorValue {
		self += 1
		return self
	}
	const IsEnded() => false
	
	Generator() => &self

	SelfSquare() => self *= self
}

func main() {
	var gen Generator = 18 // ok
	cio.Println(gen.Next) // print: 19
	if gen is int {
		gen.SelfDouble()
	} else if gen is float then gen.SelfSquare()
	// else if gen is string then ...
	// the compiler would throw an error because string don't respect the interface Generator.
	cio.Println(gen.Next()) // 39
}
